1. Los 4 Pilares de la Orientación a Objetos

Para diseñar software robusto y aplicar patrones de forma efectiva, es obligatorio dominar las cuatro propiedades fundamentales de la POO:

Abstracción

Consiste en aislar los elementos esenciales de un objeto ignorando los detalles irrelevantes de su implementación. En Java lo logramos mediante Interfaces y Clases Abstractas. Permite interactuar con un sistema sabiendo qué hace sin necesidad de saber cómo lo hace.

Encapsulamiento

Ocultamiento del estado interno de un objeto. Los atributos deben ser privados (private) y su acceso/modificación debe controlarse estrictamente mediante métodos públicos (getters/setters o métodos de negocio), garantizando que el objeto mantenga siempre un estado válido.

Herencia

Mecanismo que permite a una clase (subclase) reutilizar y extender los atributos y métodos de otra clase (superclase). En Java se aplica con la palabra clave extends.
Regla de oro: Favorece la composición sobre la herencia si la relación no cumple estrictamente un vínculo "Es Un" (Is-A).

Polimorfismo

Es la capacidad de una variable de referencia de adoptar múltiples formas dependiendo del objeto real al que apunte en tiempo de ejecución. Permite tratar objetos de diferentes subclases de manera uniforme a través de la interfaz o clase padre común.

2. Estructura de Clases y Ciclo de Vida en Java

Una clase es la plantilla o molde. El objeto es la instancia real en memoria construida a partir de ese molde.

Modificadores de Acceso (Visibilidad)

Modificador Misma Clase Mismo Paquete (Default) Subclase (Herencia) Universo (Cualquiera)
private Sí No No No
default (Ninguno) Sí Sí No No
protected Sí Sí Sí No
public Sí Sí Sí Sí

Palabras Clave Clave: static y final

  • static (Ámbito de Clase): Indica que el atributo o método pertenece a la clase en sí, no a las instancias. Existe una sola copia del atributo compartida por todos los objetos creados.
  • final (Inmutabilidad):
    • En variables: Convierte el campo en constante (no puede cambiar su valor tras asignarse).
    • En métodos: Evita que el método sea sobrescrito (@Override) por subclases.
    • En clases: Impide que la clase sea heredada (ej: la clase String en Java es final).

3. Tipos de Variables y Gestión de Memoria

En Java las variables se clasifican estrictamente en dos categorías. Comprender cómo se comportan en la memoria (Stack vs. Heap) evita errores críticos de referencias nulas y mutaciones no deseadas.

Variables Primitivas vs. Referencias (Objetos)

Característica Variables Primitivas (int, double, boolean, etc.) Variables de Referencia (Objetos, Arrays, Clases wrapper)
¿Qué almacenan? El valor real directamente en la variable. Una dirección de memoria (puntero seguro) donde reside el objeto.
Ubicación física Viven completamente en la memoria Stack. La variable (dirección) vive en el Stack, pero el objeto real vive en el Heap.
Paso por parámetro Se pasan por valor (se copia el dato, el original no se altera). Se pasa por valor la referencia (se copia la dirección. Si modificas las propiedades internas del objeto dentro del método, los cambios afectarán al objeto original).
Valor por defecto 0, 0.0 o false. null (No apunta a ninguna dirección todavía).
¡Cuidado con String e Inmutabilidad! Aunque String es un objeto en Java (Referencia), está diseñado para ser inmutable. Cualquier operación de modificación (como .toUpperCase()) no altera el String original; genera un objeto nuevo totalmente distinto en el Heap.

4. Patrón Composite (Estructural)

Propósito: Permite componer objetos en estructuras de árbol para representar jerarquías de parte-todo. Hace que los clientes puedan tratar de manera uniforme a los objetos individuales y a las composiciones de objetos.

Cómo desarrollarlo

  1. Componente (Clase base/Interfaz): Declara la interfaz común para los objetos de la composición (métodos de negocio) y, opcionalmente, para la gestión de hijos.
  2. Hoja (Leaf): Representa los objetos finales que no tienen hijos. Implementa las operaciones de negocio directas.
  3. Compuesto (Composite): Almacena componentes hijos (mediante una lista). Implementa las operaciones delegando el trabajo a sus hijos y agregando o modificando el resultado.

Qué tener en cuenta en el desarrollo

  • Transparencia vs. Seguridad: Si defines los métodos de gestión de hijos (add(), remove()) en la clase base genérica, ganas transparencia (puedes tratar a las Hojas y Compuestos exactamente igual), pero pierdes seguridad (llamar a add() en una Hoja lanzará un error en tiempo de ejecución).
  • Caché de resultados: Si el recorrido del árbol para calcular algo es costoso, considera implementar un sistema de caché en los nodos compuestos que se invalide cuando cambie la estructura.
Ejemplo clásico: Estructuras de archivos (Archivos y Carpetas), elementos de una interfaz gráfica (Botón dentro de un Panel dentro de una Ventana).

5. Patrón Visitor (Comportamiento)

Propósito: Permite separar un algoritmo o una operación de la estructura de objetos sobre la que opera. De esta forma, se pueden añadir nuevas operaciones a una jerarquía de clases existente sin modificar dichas clases.

Cómo desarrollarlo

  1. Interfaz Visitor: Declara un método visit(ElementoA), visit(ElementoB) por cada clase concreta de la jerarquía.
  2. Clase Visitante Concreta: Implementa cada método con la lógica específica del algoritmo para esa clase.
  3. Método Accept: Añadir el método public void accept(Visitor v) en la interfaz base de la jerarquía de objetos y obligar a cada clase concreta a implementarlo haciendo v.visit(this);.

Qué tener en cuenta en el desarrollo

  • El precio de la estabilidad: Este patrón es excelente si la jerarquía de clases de objetos es estable. Si añades una nueva clase a la jerarquía en el futuro, tendrás que modificar la interfaz Visitor y romperás todos los visitantes existentes.
  • Double Dispatch (Doble Despacho): Recuerda que el truco está en que accept() resuelve en tiempo de ejecución a qué clase pertenece el objeto, y la sobrecarga del método visit() mapea estáticamente el tipo en el visitante.

6. Factory Method (Creacional)

Propósito: Define una interfaz para crear un objeto, pero deja que las subclases decidan qué clase instanciar. Permite que una clase delegue la instanciación a subclases.

Cómo desarrollarlo

  1. Producto: Interfaz común para todos los objetos que el método de fábrica creará.
  2. Productos Concretos: Implementaciones específicas del Producto.
  3. Creador (Abstracto o Interfaz): Declara el método de fábrica (ej: factoryMethod()) que devuelve un objeto Producto.
  4. Creadores Concretos: Sobrescriben el método de fábrica para devolver una instancia de un Producto Concreto específico.

Qué tener en cuenta en el desarrollo

  • Acoplamiento laxo: El código cliente solo conoce la interfaz del Creador y del Producto. Esto te permite cambiar los productos concretos en el futuro sin romper el código que los usa.
  • Proliferación de subclases: El principal peligro es que para cada nuevo producto concreto que necesites, te verás obligado a crear una nueva subclase del Creador.

7. Patrón Builder (Creacional)

Propósito: Permite construir objetos complejos paso a paso. El patrón permite producir diferentes tipos y representaciones de un objeto utilizando el mismo código de construcción.

Cómo desarrollarlo

  1. Constructor (Builder Interface): Declara los pasos de construcción de los objetos comunes a todos los tipos de constructores.
  2. Constructores Concretos: Implementan los pasos de construcción y mantienen el resultado (objeto final) en su estado interno. Ofrecen un método (ej: build()) para extraer el objeto.

Qué tener en cuenta en el desarrollo

  • Inmutabilidad: Es el patrón rey para crear objetos **inmutables**. Configuras todo en el constructor intermedio y, al llamar al método final (usualmente llamado build()), se genera el objeto definitivo con parámetros de "solo lectura" (sin setters).
  • Retorno Encadenado (Fluent Interface): Asegúrate de que cada método del Builder devuelva el propio objeto Builder (return this;) para permitir el encadenamiento de métodos (ej: builder.setName("A").SetAge(10).build()).
